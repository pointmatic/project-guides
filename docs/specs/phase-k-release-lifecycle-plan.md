# Phase K: Release & Phase Lifecycle Plan

> Generated by `plan_phase` mode on 2026-04-10. This plan combines a mini-concept, feature requirements, and technical changes for Phase K. Once approved, the corresponding stories will be added to `docs/specs/stories.md`.

## Mini-Concept (Why)

Phase J shipped 22 stories (v2.0.0–v2.0.20) into a single `stories.md`, which is now ~750 lines and unwieldy to scan. The next post-release phase needs a clean slate, but the project has no built-in way to archive completed work, no codified rules for phase letters past `Z`, and no mechanism in `default` mode to suggest "what next?" once everything is `[Done]`.

Phase K closes this lifecycle gap by introducing `archive_stories` mode and the supporting metadata, template, and shared-include changes that make clean phase transitions a first-class workflow — without forcing any specific git/release flow on the developer.

## Gap Analysis

| Capability | Today | Needed |
|---|---|---|
| Archive a completed `stories.md` | None — manual file move | `archive_stories` mode that moves `stories.md` → `docs/specs/.archive/stories-vX.Y.Z.md` and writes a fresh empty `stories.md` |
| Phase letter scheme past `Z` | Undefined | Codified: `A`–`Z`, then `AA`, `AB`, …; documented in one shared include |
| `## Future` section in `stories.md` | Convention only (used in current file) | Codified into the `templates/artifacts/stories.md` artifact template; carried over by `archive_stories` |
| `default` mode "what next?" when all stories are `[Done]` | Static lifecycle table | Detect "all stories `[Done]`" and suggest two next modes: `archive_stories` or `plan_phase` |
| Action types in metadata | `create`, `modify` | Add `archive` (or compose `move` + `create`) so `archive_stories` can be declared in `.metadata.yml` |
| `plan_phase` against an empty `stories.md` | Works (file exists, just empty) but mode template assumes content to extend | Mode template handles the post-archive case: empty body, carry-over `## Future`, continue phase letter from latest archive |

## Feature Requirements (What)

### F1. `archive_stories` mode
- A new sequence-style mode invoked via `project-guide mode archive_stories`.
- Determines the archive version by reading the most recent `vX.Y.Z` from existing stories in `stories.md` (the latest version any story declares).
- Determines the latest phase letter from existing stories in `stories.md` and records it so the next phase can continue.
- Moves the current `docs/specs/stories.md` to `docs/specs/.archive/stories-vX.Y.Z.md`.
- Creates `.archive/` if it does not exist.
- Generates a fresh empty `docs/specs/stories.md` from `templates/artifacts/stories.md`, preserving the `## Future` section from the archived file.
- Suggests `plan_phase` as the natural next mode.
- Requires the developer to confirm before performing the move (approval gate).

### F2. Codified `## Future` section in `stories.md` artifact template
- The `templates/artifacts/stories.md` template includes a `## Future` heading with brief instructions on what belongs there (deferred stories, future phases, out-of-scope notes).
- An empty `## Future` section is acceptable; it does not need placeholder content.

### F3. Codified phase letter scheme
- Letters proceed `A, B, C, … Y, Z, AA, AB, … AZ, BA, …` (base-26 with letters, no zero).
- Documented in a single shared include consumed by `plan_stories`, `plan_phase`, `archive_stories`, and `refactor_plan` modes.
- The shared include also documents story sub-lettering (`A.a`, `A.b`, …, `A.z`, `A.aa`, `A.ab`, …) so the same rules apply at both levels.

### F4. `default` mode "all stories done" detection
- When the rendered `default` mode runs, it checks `docs/specs/stories.md` for the presence of any non-`[Done]` stories.
- If all stories are `[Done]` (and at least one story exists), the mode prompts the developer with two suggested next modes:
  - `archive_stories` — start the next phase with a clean slate
  - `plan_phase` — plan the next phase first, then optionally archive
- If any stories remain in any non-`[Done]` state, the existing default lifecycle suggestions apply unchanged.

### F5. `plan_phase` mode adjustments for post-archive
- Mode template explicitly handles an empty `stories.md` (no existing phases).
- When determining the next phase letter, it reads `docs/specs/.archive/` (if present) to continue from the latest archived phase rather than restarting at `A`.
- References the new shared phase-letter include rather than restating the rules.

### F6. `project-essentials.md` placeholder and render hook (added late in planning)
- A new per-project artifact `docs/specs/project-essentials.md` captures must-know facts that future LLMs need to avoid blunders (workflow rules, architecture quirks, dogfooding/meta notes).
- The render pipeline reads the file (if present) and injects it into every rendered mode via `_header-common.md` as a `## Project Essentials` section.
- Empty/missing renders cleanly with the section omitted — zero risk for new projects.
- Phase K only ships the placeholder + render hook + dogfoods it for this project. Wiring the planning modes to populate the file is deferred to **Phase L**.
- Rationale: a project-level artifact replaces the brittle "remember template source paths" pattern with a self-documenting, project-scoped guardrail that benefits every LLM that touches the project, not just one with a populated memory store.

### F7. Render output validation — fail on unrendered placeholders (added late in planning)
- After every render, the pipeline validates the output and raises `RenderError` if any `{{ identifier }}` Jinja-style placeholders remain.
- Catches: missed render.py context variables, typos in template variable names, removed `{% if %}` guards, and any other case where a Jinja variable silently fails to substitute.
- The current `_LenientUndefined` class (`render.py:83-99`) emits `{{ var_name }}` for undefined variables — historically a soft fallback to "preserve LLM instruction placeholders". F7 turns that emission into a hard error gate so unrendered placeholders never reach the LLM.
- Lenient mode itself stays unchanged because its placeholder output is what the validator scans for.
- Rationale (from developer): "No artifact should have an unrendered include. Its presence in the artifact is evidence of a missed intent or an incorrect name." A general safeguard is strictly better than per-variable assertions because it covers every template, not just the one that motivated the question.

## Technical Changes (How)

### T1. New mode template: `archive-stories-mode.md`
- Location: `project_guide/templates/project-guide/templates/modes/archive-stories-mode.md`.
- Sequence-style; uses `_header-sequence.md`.
- Steps:
  1. Read `docs/specs/stories.md`; verify all stories are `[Done]` (warn the developer if any are not).
  2. Determine archive version `vX.Y.Z` from the latest story version found.
  3. Determine the latest phase letter used.
  4. Read the existing `## Future` section content (if any).
  5. Show the developer the planned archive path and ask for confirmation.
  6. Create `docs/specs/.archive/` if missing; move `stories.md` → `.archive/stories-vX.Y.Z.md`.
  7. Render fresh `stories.md` from the artifact template, re-injecting the carried-over `## Future` section.
  8. Suggest `plan_phase` as the next mode.

### T2. New shared include: `_phase-letters.md`
- Location: `project_guide/templates/project-guide/templates/modes/_phase-letters.md`.
- Documents phase letter and story sub-letter rules (F3).
- Included by `plan-stories-mode.md`, `plan-phase-mode.md`, `archive-stories-mode.md`, and `refactor-plan-mode.md` via `{% include "modes/_phase-letters.md" %}`.

### T3. Update `templates/artifacts/stories.md`
- Add the codified `## Future` section heading and a brief inline comment block describing what belongs there.
- Section is rendered after `{{ phases_and_stories }}`.

### T4. `.metadata.yml` additions
- Add the `archive_stories` mode entry to `project_guide/templates/project-guide/.metadata.yml` under the `# Post-Release` section (the `docs/project-guide/.metadata.yml` mirror is the installed copy and is regenerated by `project-guide update` — do not hand-edit it):

  ```yaml
  - name: archive_stories
    info: Archive completed stories.md and start fresh
    description: Move docs/specs/stories.md to docs/specs/.archive/stories-vX.Y.Z.md once all stories are [Done], then create a fresh empty stories.md preserving any ## Future section.
    sequence_or_cycle: sequence
    generation_type: document
    mode_template: "{{mode_templates_path}}/archive-stories-mode.md"
    next_mode: plan_phase
    artifacts:
      - file: "{{spec_artifacts_path}}/stories.md"
        action: archive
    files_exist:
      - "{{spec_artifacts_path}}/stories.md"
  ```

- Introduce a new `archive` action type in `metadata.py` and any action handler/validator. Semantics: move the source artifact into a sibling `.archive/` directory with a versioned filename, then re-create the artifact from its template.
- Decision to confirm during implementation: whether `archive` is one atomic action or composed from `move` + `create`. The plan currently assumes a single `archive` action because the version-derivation logic is mode-specific and benefits from being co-located.

### T5. Update `plan-stories-mode.md`
- Replace the inline phase progression letter assumption with `{% include "modes/_phase-letters.md" %}`.
- Keep the recommended phase progression table; only the letter scheme moves to the include.

### T6. Update `plan-phase-mode.md`
- Add `{% include "modes/_phase-letters.md" %}`.
- Add explicit handling for empty `stories.md`: read `docs/specs/.archive/` (if present), find the latest archived `stories-vX.Y.Z.md`, parse its latest phase letter, and continue from the next letter.
- Note that an empty `stories.md` is now a valid starting state.

### T7. Update `refactor-plan-mode.md`
- Add `{% include "modes/_phase-letters.md" %}` if it touches stories renumbering. If not, skip this update — do not add unused includes.

### T8. Update `default-mode.md`
- Add a "When all stories are done" section that the LLM uses to detect the all-`[Done]` condition and suggest `archive_stories` or `plan_phase`.
- Detection is performed by the LLM at render-read time, not by the Python CLI — consistent with the v2 architecture.

### T9. Tests
- Unit tests for the new `archive` action type (file move, .archive dir creation, version derivation).
- Unit tests for `archive_stories` metadata loading and validation.
- Integration test: render the mode, perform an archive against a fixture `stories.md`, verify the new file matches the artifact template and that the `## Future` section was carried over.

### T10. Documentation
- Update the modes documentation (`docs/project-guide/modes.md` or wherever modes are catalogued) to list `archive_stories`.
- Update CHANGELOG and bump version per story.

### T11. `project-essentials.md` artifact + render context variable (F6)
- New artifact template: `project_guide/templates/project-guide/templates/artifacts/project-essentials.md` — empty body with a comment block describing what belongs there.
- Update `project_guide/render.py` to read `<spec_artifacts_path>/project-essentials.md` if present, and pass content as the `project_essentials` Jinja2 context variable (default empty string).
- **Decision:** use a render-context variable rather than a Jinja2 `{% include %}`, because the existing `FileSystemLoader` is rooted at `templates/` only and the artifact lives under `docs/specs/`. Adding the spec dir to the loader path would expose all spec files to template includes — more surface than warranted.
- Update `project_guide/templates/project-guide/templates/modes/_header-common.md` to render the variable in a `## Project Essentials` section between the Rules block and the mode heading. Use `{% if project_essentials %}...{% endif %}` so empty/missing files produce no output (no empty header, no extra blank lines).
- **Why the `{% if %}` guard matters**: `render.py:83-99` defines `_LenientUndefined`, which renders undefined variables as the literal string `{{ var_name }}`. If someone wrote a bare `{{ project_essentials }}` without the guard and render.py forgot to set the variable, the LLM would see an ugly literal `{{ project_essentials }}` in the rendered output. The `{% if %}` guard makes this safe under both conditions: empty string is falsy, and `_LenientUndefined.__bool__` also returns `False`.
- Tests in `tests/test_render.py` for present-non-empty / present-empty / absent, plus a regression-guard assertion that the literal string `{{ project_essentials }}` never appears in any rendered output.
- Dogfood for this project: populate `docs/specs/project-essentials.md` with the dogfooding rule, the `pyve run` rule, and the "go.md is regenerated" rule.
- **Out of scope here**: wiring the planning modes to populate the file. That is Phase L's entire purpose.

### T12. Render output validator (F7)
- Add a private validator function in `project_guide/render.py` that runs after `template.render(...)` and before `output_path.write_text(...)`.
- Regex: `\{\{\s*[a-zA-Z_]\w*\s*\}\}` — matches single-identifier Jinja placeholders. Does not attempt to match complex expressions (`{{ obj.attr }}`, `{{ var | filter }}`) because `_LenientUndefined.__str__` reduces those to the bare identifier form anyway.
- On match, raise `RenderError` with the offending placeholder names and a hint pointing at (a) render.py context variables and (b) template variable spellings.
- Tests: undefined variable raises, defined variable succeeds, validator no-ops on templates with no placeholders.
- Audit current mode templates for any that silently rely on undefined variables; fix any found.
- **Known limitation, documented in code**: templates that legitimately want to emit a literal `{{ var }}` string (e.g., Jinja documentation) will trigger a false positive. Not a current concern; bridge if/when it surfaces.
- **Interaction with F6 (K.h)**: K.h's one-off "rendered output never contains `{{ project_essentials }}`" assertion becomes redundant when T12 ships in K.i. K.i's tasks include removing the K.h-specific assertion and leaving a comment pointing at the generalized validator. Until K.i ships, the K.h assertion provides immediate protection.

## Out of Scope for Phase K

- Release helper / version-bump / tag automation — explicitly deferred (developer works across pure-git, GitHub-tag-triggered, and branch-flow setups and prefers to stay tool-agnostic for now).
- `code_production` mode and `code_*` abstraction (Test First flag) — deferred to a separate phase focused on the code mode hierarchy.
- Audit modes (`audit_security`, `audit_architecture`, etc.) — deferred.
- Automatic detection of "release just shipped" beyond the all-stories-`[Done]` heuristic.
- Migration tooling for old `stories.md` files that predate the `## Future` convention — assume existing files are well-formed or get fixed manually.
- A `status` command flag to trigger archive — not needed; the mode itself is the entry point.
